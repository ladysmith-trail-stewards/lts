# DX Improvements: db:studio, db:reset Formatting, POLICIES.md Stability

## Flags

| Flag | |
|---|---|
| DB Change | ⬜ |
| Style Only | ⬜ |
| Env Update Required | ⬜ |

## Problem

Three small developer experience issues found while working on the Google SSO feature:

1. `pnpm db:studio` ran `supabase studio`, a command that no longer exists in the Supabase CLI — it errored immediately.
2. `pnpm db:reset` ran Prettier verbosely, printing every reformatted filename and making it hard to spot real warnings in the output.
3. `supabase/POLICIES.md` included an ISO timestamp in its header, meaning the file was always dirty in git after any `db:reset` run — even if the schema hadn't changed.

## Changes

### `db:studio` command

`package.json` `db:studio` script changed from:

```
"db:studio": "supabase studio"
```

to:

```
"db:studio": "open http://127.0.0.1:54323"
```

Studio is always running at port 54323 when `supabase start` is active.

### `db:reset` Prettier step

`scripts/db-reset.js` now runs Prettier quietly and prints only a summary line:

```
▶ Format source files
  reformatted 3 file(s)
```

Uses `--list-different` to count changed files before silently writing them with `--log-level silent`.

### `POLICIES.md` timestamp removed

`scripts/extract-db-policies.js` no longer writes the generation timestamp to `supabase/POLICIES.md`. The header is now static:

```md
> Auto-generated by `scripts/extract-db-policies.js`.
```

This makes POLICIES.md stable between runs when the schema is unchanged, keeping it clean in git diffs.
